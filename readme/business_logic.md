# Logistics CRM: Комплексна Архітектура та Бізнес-Логіка

Цей документ є єдиним і абсолютним джерелом правдивості (Source of Truth) щодо бізнес-правил, зв'язків між сутностями та архітектурних патернів CRM-системи.

## 👥 1. Управління Доступом на основі Ролей (RBAC) та Видимість (Scoping)

Додаток базується на моделі B2B, де `Користувачі (Users)` належать до `Компаній (Companies)`. Усі критичні дії суворо розмежовані на три ролі:

### 👑 Адміністратор (Перевізник / Власник Платформи)
- **Ідентифікація:** Представляє власника логістичного парку, який надає послуги перевезення (Admin транспортної компанії).
- **Можливості:**
  - Абсолютний доступ до читання та запису всього (політика `viewAny` повертає `true`).
  - Управління фізичним автопарком (`Trucks`) та глобальне перевизначення бізнес-налаштувань (`Tariffs`, `VehicleTypes`).
  - **Замовлення:** Може створювати замовлення від імені *будь-якої* зареєстрованої компанії-клієнта. Має право проводити замовлення через весь його життєвий цикл (Нове -> В обробці -> В дорозі -> Доставлено -> Скасовано).
  - **Єдина роль, яка має право призначати фізичну вантажівку (`Truck`) на конкретне замовлення (`Order`).**

### 🏢 Менеджер (Представник Клієнта / Логіст-Замовник)
- **Ідентифікація:** Представник компанії-клієнта (наприклад, "Агро-Транзит ТОВ"), що замовляє послуги.
- **Можливості:**
  - Суворо обмежений переглядом даних **лише** своєї компанії (по `company_id`).
  - **Замовлення:** Може створювати нові замовлення для своєї компанії та переглядати всі замовлення компанії.
  - **Користувачі:** Наразі не має доступу до перегляду чи управління іншими користувачами компанії (це робить Адміністратор платформи).
  - **Скасування:** Має право скасувати замовлення (`status: canceled`), але **СТРОГО ЛИШЕ** якщо замовлення наразі перебуває у статусах `new` (нове) або `pending` (в обробці). Як тільки адмін призначив вантажівку і статус змінився (`in_transit` або завершено), менеджер втрачає право на редагування замовлення.

### 👷 Спостерігач (Observer) (Працівник Складу Клієнта / Логіст-Координатор)
- **Ідентифікація:** Працівник компанії-клієнта (наприклад, вантажник або координатор на складі).
- **Можливості:**
  - **Видимість:** Має абсолютно такий самий обсяг видимості, як і Менеджер. Він може бачити **всі замовлення**, що належать до його `company_id` (щоб розуміти, які вантажівки сьогодні приїдуть на завантаження).
  - **Замовлення:** **НЕ МОЖЕ** створювати нові заявки/замовлення.
  - **Модифікації:** На 100% заблоковано. Не має права створювати, редагувати, скасовувати або змінювати статус *жодного* замовлення. Це суто спостерігач.

## 🗄️ 2. Основні Сутності та Архітектура Автопарку

### Шар Абстракції Автопарку (`VehicleType` проти `Truck`)

Щоб уникнути створення надскладної системи планування (Gantt-графіків), але при цьому зберегти реалістичні логістичні обмеження, CRM абстрагує замовлення від фізичного заліза:

1. **`VehicleType` (Тип Транспорту - Запит):** Приклади: "Спринтер (до 2т)" або "Мега Тент (24т)". 
   - Клієнти (Менеджери / Спостерігачі) взаємодіють **лише** з Типами Транспорту. Вони запитують *клас* автомобіля.
   - Типи Транспорту містять параметр `base_price_per_km_cents`, який слугує для попередньої оцінки вартості (upfront cost).

2. **`Truck` (Вантажівка - Фізична Реальність):** Реальний фізичний транспортний засіб із номерним знаком (наприклад, `АА 1234 ВВ`).
   - З фізичними вантажівками взаємодіють **лише Адміністратори**.
   - **Правило Призначення (`AssignTruckAction`):** Адміністратор може призначити `Truck` на Замовлення **тільки якщо**:
     1. Статус замовлення наразі `new` або `pending` (не доставлено чи не скасовано).
     2. Фізична вантажівка `Truck` належить до **того самого** `vehicle_type_id`, який запросив Менеджер клієнта при оформленні замовлення.
     3. Фізична вантажівка є **Вільна (Available)**. Вантажівка є суворо недоступною для призначення, якщо вона вже прив'язана до іншого замовлення, яке наразі знаходиться у статусі `in_transit` (в дорозі) або `loading` (завантаження).

### Замовлення (`Order`) та Вантаж (`OrderItem`)
- `Order` представляє собою повне перевезення з Точки А (Завантаження) до Точки Б (Розвантаження).
- Містить геопросторові поля для мапи (`pickup_lat`, `pickup_lng`, `delivery_lat`, `delivery_lng`, `distance_km`).
- Одне Замовлення має багато `OrderItem` (Позицій Вантажу), що містять точну вагу (`weight_kg`), розміри (`cbm`) та грошову цінність (`declared_value_cents`).

## 🏗️ 3. Двигун Оцінки Вантажів (Архітектура Пайплайнів / PricingPipeline)

Система уникає громіздких, монолітних функцій ціноутворення, використовуючи суворо типізований патерн `Pipeline` з Laravel (`App\Logistics\Cargo\PricingPipeline`). 

Коли створюється замовлення, дані проганяються через ізольовані, послідовні кроки ("Труби" / Pipes):

1. **`CalculateBasePrice`:**
   - Множить кілометраж `distance_km` на тариф `base_price_per_km_cents` запитаного типу машини `VehicleType`.
   - *Динамічний Fallback:* Якщо клієнт не обрав конкретного типу транспорту, система бере найвищий середній тариф для покриття потенційних ризиків. Якщо згодом Адмін призначить дешевшу/меншу машину, ціна динамічно зменшиться (з відповідним логом в історії).
2. **`ApplyDangerousGoodsSurcharge`:**
   - Перевіряє глобальні налаштування `Tariff`, і, якщо бодай одна позиція вантажу є "небезпечною" (`is_dangerous`), нараховує відсоткову націнку ADR (наприклад, 25% від бази).
3. **`ApplyInsuranceFee`:**
   - Сумує всю задекларовану вартість вантажу (`declared_value_cents`) і розраховує обов'язковий відсоток страхування з налаштувань (наприклад, 1%).
4. **`ApplyVolumeDiscount`:**
   - (Опційне правило) Надає automatic знижку, якщо загальний об'єм (CBM) перевищує певну межу, стимулюючи великі замовлення.
5. **`ApplyTax`:**
   - Бере проміжний підсумок (База + Націнки + Страховка - Знижки) і нараховує глобальний ПДВ (VAT) із налаштувань (зазвичай 20%).

*Увага: Всі фінансові транзакції розраховуються та зберігаються у **центах / копійках** як Цілі числа (Integers), щоб повністю ліквідувати проблеми з точністю чисел із рухомою комою (floating-point precision errors).*

## ⚙️ 4. API та Зовнішні Сервіси

1. **`RoutingService` (OSRM / Haversine):**
   - Рахує реальну дистанцію дорогами (`distance_km`) та орієнтовний час у дорозі (`transit_time_minutes`) між координатами завантаження і розвантаження.
   - Результати **автоматично кешуються у Redis**, використовуючи унікальний хеш маршруту (на основі точних координат). Це допомагає обійти Rate Limits зовнішніх API та миттєво рахувати популярні зворотні рейси (напр. Київ <-> Одеса).
   - У випадку падіння зовнішнього API OSRM автоматично вмикається "Fallback" - прораховує дистанцію по прямій лінії за математичною формулою Наверсіна (Haversine formula).

2. **`DashboardService`:**
   - Розраховує загальні агреговані метрики (Загальний Дохід, Активні Компанії, Останні Замовлення).
   - Суворо застосовує фільтри RBAC (див. Розділ 1): Адміністратори бачать глобальні прибутки платформи, Клієнти (і Менеджери, і Спостерігачі) бачать виключно статистику та витрати своєї власної компанії.
